# 한국어 띄어쓰기 조회 도구 계획서

## 1. 한 줄 요약

단어나 표현을 넣으면 **우리말샘 사전 정보와 관련 맞춤법 조항을 한자리에 펼쳐**, 그것이 한 단어로 등재돼 있는지·어떤 품사인지·어느 조항이 걸리는지를 보여 주는 도구. 글을 쓰다 띄어쓰기가 헷갈릴 때 사전을 따로 찾는 수고를 도구 안에서 해결한다.

---

## 2. 배경과 목적

띄어쓰기는 한글 맞춤법 제2항("문장의 각 단어는 띄어 씀을 원칙으로 한다")을 기본으로 하지만, 실제 판단은 **그 표현이 사전에 한 단어로 등재돼 있는지**와 **그 글자가 어떤 품사로 쓰였는지**에 좌우된다. "큰집"(종갓집·맏집)은 한 단어라 붙여 쓰지만 "크기가 큰 집"이면 띄어 쓴다. "대"는 의존명사면 띄우고(차 한 대) 접사면 붙인다(대국민 담화).

목표는 **글을 쓸 때마다 우리말샘 사이트를 직접 뒤지는 수고를 없애는 것**이다. 도구는 사전·규정 정보를 펼쳐 보여 주고, 띄어쓰기 결정은 사람이 한다.

## 3. 핵심 성격 — 무엇이고 무엇이 아닌가

| 이 도구는 | 이 도구가 아닌 것 |
|---|---|
| 정보 제공 도구 | 자동 교정기 |
| 판단 근거(사전·품사·조항)를 펼쳐 줌 | "이렇게 고치세요"라고 단정함 |
| 품사 판단을 사전에 위임 | 품사를 스스로 추정·재분류 |
| 최종 판단을 사람에게 맡김 | 결과를 사용자에게 강요 |

이 성격이 설계 전반의 기준점이다. "틀렸다"고 말하지 않기 때문에, 동형이의어처럼 도구가 판별할 수 없는 경우에도 사용자를 오도하지 않는다 — 후보를 정리해 보여 줄 뿐이다.

---

## 4. 동작 엔진 — 품사가 띄어쓰기의 분기 조건

이 도구의 심장은 단순하다.

```
우리말샘 조회 → 품사(pos) 추출 → 품사를 조항에 매핑 → 띄어쓰기 안내
```

띄어쓰기 규칙은 사실상 **품사로 갈린다.** 조사는 붙이고(제41항), 의존명사는 띄우고(제42·43·45항), 보조용언은 띄움 원칙/붙임 허용(제47항), 접사는 붙인다. 따라서 "이 글자가 어떤 품사로 쓰였나"가 정해지면 띄어쓰기는 거의 자동으로 따라온다.

**품사는 우리말샘이 권위 있게 알려 준다.** API 응답의 품사 필드에 "명사·의존명사·조사·보조동사·접사" 등이 들어 있으므로, 형태소 분석으로 추정할 필요가 없다. 이것이 kiwipiepy를 핵심에서 뺀 근거다.

### 4.1 품사 → 조항 매핑 테이블

도구의 조회용 색인. 사전이 품사를 주면 이 표로 조항을 찾고, 조항 번호로 조항 데이터(원문·예시·해설)를 꺼낸다.

| 우리말샘 품사 | 보정 단서 | 적용 조항 | 띄어쓰기 |
|---|---|---|---|
| 조사 | — | 제41항 | 붙임 |
| 어미 | — | (제41·47항 참고) | 붙임 |
| 의존명사 | 기본 | 제42항 | 띄움(원칙) |
| 의존명사 | 뜻풀이에 '세는 단위' / 앞에 수사 | 제43항 | 띄움 / 숫자·순서 뒤 붙임 허용 |
| 의존명사·부사 | 글자가 {겸·내지·대·및·등·등등·등속·등지·따위} | 제45항 | 띄움 |
| 보조동사·보조형용사 | — | 제47항 | 띄움(원칙) / 붙임 허용 + 제약 |
| 명사(고유) | 인명 + 호칭/관직 | 제48항 | 성명 붙임 / 호칭 띄움 |
| 명사(고유) | 고유명사 연쇄 | 제49항 | 띄움(원칙) / 단위별 붙임 허용 |
| 명사 | 전문용어 표지 | 제50항 | 띄움(원칙) / 붙임 허용 |
| 수사 | 수사 연쇄 | 제44항 | 만 단위 띄움 |
| 접사(접두·접미) | — | (제2항 외) | 붙임 |
| 명사(일반) | 한 단어로 등재됨 | 제2항 | 등재형 그대로 / 아니면 띄움 |

**보정은 재분류가 아니다.** 사전 품사를 덮어쓰지 않고, "이 품사에 이 단서가 더해지면 이 조항도 함께 안내"하는 방식이다. 보정이 필요한 곳은 두 군데뿐이다.

1. **단위명사(제43항) vs 일반 의존명사(제42항)** — 우리말샘이 둘 다 "의존명사"로 줄 때가 많아, 뜻풀이('세는 단위')나 앞 수사로 가린다.
2. **열거하는 말(제45항)** — 닫힌 목록({겸·내지·대·및·등 …})에 글자가 들면 45항을 함께 안내한다.

### 4.2 사전으로 갈리지 않는 경우 (정직한 한계)

**제42항 동형이의어**는 사전 품사로도 갈리지 않는다. "뿐/대로/만큼/만/지/차/들/판"은 같은 글자가 조사·어미·접사일 때도, 의존명사일 때도 있는데, **앞말이 체언이냐 용언 관형사형이냐**라는 문맥으로만 결정된다. 우리말샘은 두 표제어를 모두 갖고 있지만 "지금 이 자리에서 어느 쪽"은 정해 주지 못한다.

→ 이것은 보정의 대상이 아니라 **"판별하지 않음" 원칙의 영역**이다. 도구는 **양쪽 가능성을 함께 안내**한다.

| 글자 | 붙임(조사·어미·접사 등) | 띄움(의존명사) |
|---|---|---|
| 뿐 | 남자뿐이다 | 웃을 뿐이다 |
| 대로 | 법대로 | 아는 대로 |
| 만큼 | 전봇대만큼 | 볼 만큼 |
| 만 | 하나만 | 떠난 지 사흘 만에 |
| 지 | 큰지 작은지 (어미) | 떠난 지 오래다 |
| 차(次) | 인사차 (접사) | 가려던 차에 |
| 들 | 학생들 (접사) | 쌀, 보리 들 |
| 판 | 노름판 (합성어) | 바둑 두 판 |

## 5. 입력과 출력

### 5.1 입력

- **주 입력**: 단어 또는 짧은 표현 하나 ("큰집", "할 만하다", "차 한 대", "대")
- 붙여 쓴 덩어리가 섞여 있어도 처리 ("차 한대" → 분절 안내)
- **문장 단위 입력은 향후 확장**(8장). 지금은 다루지 않는다.

### 5.2 출력 — 두 층 구조

어떤 입력이든 결과 카드는 항상 두 층을 같은 자리에 둔다. 사전이 답을 주는 경우든 규정이 답을 주는 경우든 눈이 헤매지 않게 한다.

- **사전 영역** (흰 바탕): 표제어·품사·뜻풀이, 그리고 **뜻마다 띄어쓰기 직결 안내**
- **규정 안내 영역** (회색 구분): 적용 조항·원칙/허용, 사전 등재와 무관함을 표시

케이스에 따라 무게중심만 옮긴다. (시안은 7장 참고)

### 5.3 표시 규칙

- **별개 표제어(동형이의어)는 누락 없이 모두 표시.** 단, 단음절처럼 표제어가 많으면 **품사별로 묶어 압축**한다(아래).
- **한 표제어 안의 다의는 3개까지 펼치고**, 나머지는 "외 N개 뜻"으로 접는다.
- **의미 번호는 빈도·중요도 순서가 아님**을 항상 주석으로 명시한다. (선별이 아니라 분량 조절이며, 도구가 "어느 뜻이 중요한지" 고르지 않는다는 철학과 일치.)
- **뜻↔띄어쓰기 직결**: 사전 뜻마다 "이 뜻이면 붙여 씀 / 띄어 씀"을 붙인다. 이것이 사용자가 가장 원한 기능이다.

### 5.4 단음절 대응 — 품사별 압축

단음절은 표제어가 폭발한다("대"는 표제어만 9개+). 뜻을 줄세우지 않고 **품사로 묶는다.** 띄어쓰기 결정에 필요한 건 뜻이 아니라 품사이기 때문이다.

```
대  [표제어 9개 · 품사 4종]
 ├ 의존명사     — 차·기계를 세는 단위      → 띄어 씀 (제43항)
 ├ 명사(표제어 5) — 대(臺)·대(代)·대(對)…   → 한 단어, 그대로  [펼치기]
 ├ 접사(표제어 2) — 대-(접두) 대항하는       → 붙여 씀
 └ 의존명사(열거) — 청군 대 백군            → 띄어 씀 (제45항)
```

품사 블록마다 오른쪽에 띄어쓰기 배지를 붙인다. 사용자는 "내가 쓰려는 게 단위(대)인지 마주함(대)인지"만 떠올리면 답이 나온다. 뜻풀이가 궁금하면 그때 펼친다. **도구는 어느 뜻인지 고르지 않는다.**

### 5.5 분절 안내 블록

붙여 쓴 덩어리("차 한대")가 들어오면, 결과 카드 맨 위에 **나눠질 수 있음을 알리는 블록**을 띄운다.

```
✂ 나눠질 수 있는 덩어리
  한대 → 한(수 관형사) + 대(단위 의존명사)
  '대'가 단위 의존명사로 쓰이면 제43항에 따라 띄어 씀 → "차 한 대"
  단, 의도한 뜻을 아래에서 확인하세요.
```

말투에 주의한다 — **"틀렸다"고 하지 않는다.** "나눠질 수 있다 / 이 경우 …가 됩니다 / 의도한 뜻을 확인하세요"로 제시하고, 최종 판단은 사람에게 남긴다. 사용자가 정말 다른 뜻(예: 명사 대(臺))을 의도했을 수도 있기 때문이다.

**분절 처리 방식 (1단계는 단순하게):** kiwipiepy 없이, 덩어리의 끝 음절을 떼어 사전에 있는지 확인하는 가벼운 규칙으로 시작한다. 안 잡히면 조용히 넘어간다("애매하면 침묵"). 정확도가 부족하면 그때 형태소 분석기 도입을 검토한다(8장).

## 6. 기술 설계

### 6.1 아키텍처 — 로직과 껍데기 분리

핵심은 **두뇌(core)와 껍데기(shell)를 나누는 것**이다. core를 한 번 잘 만들면 껍데기만 바꿔 끼울 수 있다.

```
core/                         ← 사전 조회 + 품사 매핑 + 규정 안내. 한 번만 잘 만든다.
  ├ dict_client   우리말샘 API 호출·파싱
  ├ pos_mapper    품사 → 조항 매핑(4.1 표) + 단위·열거 보정
  ├ rules_data    조항 데이터(원문·예시·해설) — 별도 JSON
  ├ segmenter     분절 안내(끝 음절 분리, 단순 규칙)
  └ inspect()     입력 → 결과 데이터(dict) 반환. print/GUI 일절 없음.

shell/
  ├ gui     core.inspect() 호출 → 화면에 그림 → PyInstaller로 exe 빌드  ← 지금
  └ mcp     core.inspect() 호출 → JSON 반환 → FastMCP 서버              ← 나중
```

**철칙**: `core.inspect()`는 절대 `print`나 GUI를 직접 건드리지 않고 **결과를 데이터로 반환만** 한다. 그래야 GUI는 그 데이터를 화면에 그리고, MCP는 같은 데이터를 JSON으로 내보낸다.

### 6.2 왜 exe 먼저, MCP 나중인가

| | 단독 exe (GUI) | MCP 서버 |
|---|---|---|
| 어떻게 쓰나 | 글 쓰다 창 띄워 단어 입력 | AI 클라이언트 안에서 조회 |
| "사전 찾는 수고 없이" 목표 | **글 흐름에서 바로 조회** | AI 대화 중이어야 편함 |
| 배포 | 더블클릭 | 클라이언트 설정 필요 |

목표가 "평소 글 쓸 때"이므로 **단독 도구가 더 맞는다.** exe를 먼저 만들며 `inspect()`의 입출력 형태를 다듬으면, 그게 곧 MCP 도구의 명세가 된다. 거꾸로 하면 GUI 붙일 때 출력 형식을 다시 손봐야 한다. **둘 중 하나를 버릴 필요는 없다** — core가 공용이므로 나중에 MCP 껍데기만 추가하면 된다.

### 6.3 스택

- **언어/런타임**: Python
- **사전**: 우리말샘(opendict) 오픈 API
  - 엔드포인트: `https://opendict.korean.go.kr/api/search`
  - 요청: `key`, `q`, `req_type=json`, `advanced=y`, `method=exact`, `num`
  - 응답: `channel.item[]` — 동형이의어는 별개 `item`, 한 단어 내 다의는 `sense[]`. **품사는 각 item의 품사 필드.** (우리의 "표제어 모두 / 다의 3개" 규칙과 자연스럽게 맞물림)
  - 인증키: 환경변수 `KOREAN_DICT_API_KEY` (사용자가 직접 발급)
- **GUI**: 표준 라이브러리 우선(Tkinter) 또는 가벼운 웹뷰 — 1단계에서 결정
- **exe 빌드**: PyInstaller (`--onefile`)
- **(나중) MCP**: FastMCP, stdio 트랜스포트
- **형태소 분석**: 1단계에서 **사용 안 함**. 향후 확장 시 kiwipiepy 검토.

### 6.4 조항 데이터

별도 JSON으로 둔다. 이미 정리된 `띄어쓰기_조항_정리.json`(제2항 + 제5장 제41~50항)을 그대로 활용하며, 품사 매핑을 위해 각 항에 **적용 품사 필드**를 더한다. 규정 문구를 고칠 때 이 JSON만 건드리면 되고, 매핑 테이블(4.1)은 품사→조항 번호만 알려 주는 색인으로 분리한다.

### 6.5 처리 흐름

```
입력 (단어/표현)
  → 붙은 덩어리면 분절 후보 생성 (끝 음절 분리, 단순 규칙)
  → 우리말샘 API 조회 (동형이의어·다의 수집)
  → 품사 추출 → 매핑 테이블로 조항 결정 (단위·열거 보정 적용)
  → 결과 구성
       · 사전 카드: 표제어 모두 / 다의 3개 펼침 + 접기
       · 단음절 등 표제어 많으면 품사별 압축
       · 뜻마다 띄어쓰기 직결 안내
       · 규정 안내: 적용 조항 + 원칙/허용 (제47항 제약 포함)
       · 제42항 동형이의는 양쪽 안내
       · 붙은 덩어리면 분절 안내 블록 추가
  → 공통 주석 첨부 (판단은 사람이 / 번호는 중요도 아님 / 제2항)
  → 데이터(dict) 반환  ← GUI·MCP가 각자 표현
```

## 7. 화면 시안 (합의된 네 패턴)

도구의 모든 표시 패턴은 네 케이스로 정리된다. 케이스마다 두 층(사전·규정)의 무게중심만 옮긴다.

### 7.1 큰집 — 사전 주역 (등재 여부가 곧 띄어쓰기)

사전 영역이 주역. 표제어 "큰집"의 뜻 3개를 펼치고 각 뜻에 "이 뜻이면 붙여 씀 → 큰집"을 붙인다. 나머지는 "외 1개 뜻"으로 접는다. 규정 영역은 한 줄 보조: "단순히 크기가 큰 집을 뜻하면 표제어가 아니므로 제2항에 따라 '큰 집'."

### 7.2 할 만하다 — 규정 주역 (제47항, 원칙·허용 둘 다)

사전 카드 없음. 규정 영역이 주역. 원칙(할 만하다)·허용(할만하다)을 좌우 카드로 나란히 두고 "둘 다 맞습니다"를 명시. 아래에 **붙임 제약 경고**: "앞말에 조사가 붙거나(겨룰 만은 하다) 중간에 조사가 들면 띄어 써야 함."

### 7.3 차 한 대 — 사전+규정 혼합 (제43항)

단어별 사전 정보("대" 의존명사 = 단위 / "차" 명사)와 제43항을 함께. 원칙(차 한 대)·허용(차 1대, 숫자 뒤 붙임)을 좌우 카드로. 동형이의어("대"의 다른 표제어)는 "더 보기"로 접는다.

### 7.4 대 — 단음절 품사 압축

표제어 9개를 품사 4블록(의존명사/명사/접사/의존명사-열거)으로 묶고, 블록마다 띄어쓰기 배지. 뜻풀이는 펼치기. 맨 아래: "의도한 품사를 찾아 적용하세요" — 판별하지 않음.

### 7.5 (변형) 차 한대 — 분절 안내

7.3에 **분절 안내 블록**(5.5)이 맨 위에 추가된 형태. "한대 → 한 + 대"를 보여 주고 제43항 적용 시 "차 한 대"가 됨을 안내. "틀렸다"가 아니라 "나눠질 수 있다"로 제시.

> 시안 이미지는 대화 중 별도로 렌더링해 확인했다. 구현 시 이 네(다섯) 패턴을 기준 삼는다.

## 8. 단계별 로드맵

### 1단계 — 준비 (0.5일)
- 우리말샘 오픈 API 인증키 발급 (회원 가입 → 오픈 API 사용 신청)
- 개발 환경 구성, 의존성 확인(`httpx`, `pydantic`, GUI 라이브러리, `pyinstaller`)
- 사전 API 응답 구조 실제 호출로 검증 (동형이의어·다의·**품사 필드** 형태 확인)
- GUI 라이브러리 결정 (Tkinter vs 가벼운 웹뷰)

### 2단계 — core: 사전 조회 + 품사 매핑 (1.5~2일)
- `dict_client`: 우리말샘 호출·파싱 (인증·타임아웃·동시 호출 상한·에러 구분)
- `rules_data`: 조항 JSON에 적용 품사 필드 추가
- `pos_mapper`: 품사 → 조항 매핑(4.1) + 단위·열거 보정
- `inspect()`: 입력 → 결과 데이터(dict) 반환. **GUI/print 없이** 단위 테스트
- 대표 입력으로 검증: "큰집" "할 만하다" "차 한 대" "대" "법대로/아는 대로"

### 3단계 — core: 표시 데이터 구성 (1~1.5일)
- 표제어 모두 / 다의 3개 펼침 + 접기
- 단음절 품사별 압축
- 뜻↔띄어쓰기 직결 안내
- 제42항 동형이의 양쪽 안내, 제47항 붙임 제약 식별
- 분절 안내(끝 음절 분리, 단순 규칙) — 안 잡히면 침묵
- 공통 주석 3종

### 4단계 — shell: GUI + exe (1.5~2일)
- core.inspect() 호출 → 두 층 카드로 렌더링 (7장 패턴)
- 입력창·예시 칩·결과 카드
- 에러 메시지를 행동 지향적으로 (인증키 오류·한도 초과·타임아웃 구분)
- PyInstaller로 단일 exe 빌드 (`--onefile`), 실행 확인

### 5단계 — 문서화·배포 (0.5일)
- README (설치·인증키 발급·실행·한계)
- 인증키 입력 방법 안내 (환경변수 또는 최초 실행 시 입력)

**예상 총 소요**: 약 5~6.5일 (1인 기준, 테스트 포함)

### (이후) MCP 껍데기 추가 — 선택
- FastMCP 도구 등록: `inspect_word(word, response_format)` — core.inspect() 그대로 호출
- 어노테이션: `readOnlyHint=true`, `openWorldHint=true`(외부 사전 호출)
- MCP Inspector로 검증, 클라이언트 설정 예시(Claude Desktop 등)

## 9. 한계와 대응 (정직하게 명시)

- **도구는 판별하지 않는다.** 같은 글자가 여러 품사로 등재되면(동형이의어) 후보를 모두 펼칠 뿐, "이 뜻이다"라고 고르지 않는다. 최종 선택은 사용자. — 이는 한계가 아니라 설계 선택이다.
- **제42항 동형이의(뿐/대로/만큼/만/지/차/들/판)는 사전으로 안 갈린다.** 문맥으로만 결정되므로 양쪽을 함께 안내한다.
- **분절은 1단계에서 단순 규칙.** 끝 음절 분리로 안 잡히는 덩어리는 조용히 넘어간다. 정확도가 부족하면 형태소 분석기 도입을 검토.
- **사전에 없는 활용형·표제어**는 "찾지 못함"으로 표시. (용언은 '-다' 기본형으로 복원해 조회)
- **제49·50항(고유명사·전문용어)의 '단위' 판단**은 직관에 의존해 자동 식별이 어렵다. 조항과 원칙/허용을 안내하는 데 그친다.

## 10. 향후 확장 여지 (선택)

- **문장 단위 입력**: kiwipiepy 형태소 분석을 붙여 문장을 분절하고 어절별로 조회. (1단계에서 뺀 기능을 여기서 복원)
- 분절 정확도 개선: 단순 규칙 → 형태소 분석기
- 표준국어대사전 병행 조회 + 사전 소스 선택 옵션
- 사용자 사전 등록 (신조어·고유명사 정확도 개선)
- MCP 껍데기 정식 추가 (8장 이후 단계)

## 11. 합의된 설계 결정 요약

| 항목 | 결정 |
|---|---|
| 도구 성격 | 정보 제공(자동 교정 아님), 판단은 사람이 |
| 주 입력 | 단어·표현 하나 (문장은 향후 확장) |

| 동형이의어 | 판별 안 함. 표제어 모두 표시 |

| 뜻 표시 | 표제어 모두 + 다의 3개 펼침 / 뜻↔띄어쓰기 직결 |

| 제42항 동형이의 | 양쪽 가능성 함께 안내 |
| 분절 | 단순 규칙으로 시작, 안 잡히면 침묵 |
| 아키텍처 | core(조회+매핑) / shell(gui·mcp) 분리 |
| 배포 | 단독 exe 먼저(PyInstaller), MCP 나중 |
| 사전 키 | 환경변수 또는 최초 실행 입력 |
